      Detailed Decision Policy (when layers combine vs when they do not)
      This is the exact policy used by the optimized panel. It is deterministic and query-type based.
      A) Road-token query (combine Name Match + Address)
      Trigger if query contains a recognized road token: st/street, ave/avenue, blvd/boulevard, rd/road, dr/drive, ct/court, ln/lane.
      Behavior:
      • Run Name Token Match and Address Index together
      • Merge + dedupe results
      • Show reminder: projection used (ave <-> avenue, etc.)
      B) Non-road query (precision-first, no early combine)
      Examples: apple, field, renovaton
      Behavior:
      • Step 1: Name Token Match first
      • If Step 1 hits, stop (address not executed)
      • If Step 1 misses, run Address Index
      • If still no hit and query length >= 4, run Fuzzy (Damerau-Levenshtein)
      • If still no hit, show Fallback (recently visited + recently updated)
      C) Why this split exists
      • Road tokens are ambiguous and valid in both names and addresses, so combine is required.
      • General text queries are usually high-precision on names; early combine would add noise.
      • This keeps precision high while preserving recall for address-style queries.
      Quick examples
      • apple - Name token hit (if hit), Address skipped
      • ave - Name + Address combined
      • tro - Name token hit in retrofit
      • king street - Address projection path
      • feild - Name miss -> Fuzzy hit (transposition typo)
      • zzzz - all miss -> Fallback list
